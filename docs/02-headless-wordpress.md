# 02. Headless WordPressの概念と必要技術

## Headless WordPressとは

WordPressを「**データの倉庫（バックエンド）**」としてだけ使い、表示部分（フロントエンド）を別の技術で構築する手法。

WordPressの管理画面で記事や画像を管理しつつ、訪問者に表示されるサイトは Next.js / Nuxt / Astro などのモダンフレームワークで作る。データのやり取りはAPI経由（WP REST API または WPGraphQL）。

「Headless（首なし）」とは、データを保存する「身体」だけを使い、画面を描画する「頭」を切り離して別技術で作る発想を指す。

## 通常のWordPressとの違い

```
【通常のWordPress（モノリシック）】
管理画面 ─→ WordPressで描画 ─→ ユーザーに表示
                （PHPテンプレート）

【Headless WordPress】
管理画面 ─→ WP REST API ─→ Next.js等で描画 ─→ ユーザーに表示
            （JSONで配信）   （Reactベース）
```

| 比較項目 | 通常のWordPress | Headless WordPress |
|---------|----------------|---------------------|
| 表示技術 | PHPテンプレート | React/Vue/Astro等 |
| データ取得 | データベース直接 | API経由（JSON） |
| 表示速度 | 通常 | 圧倒的に速い |
| デザイン自由度 | テーマに依存 | 制限なし |
| 編集者の使い勝手 | 一体型で直感的 | プレビュー実装が必要 |
| 構築コスト | 低い | 高い（2〜3倍の工数） |
| 必要な技術 | PHP中心 | JavaScript/React中心 |

## Headlessの本当の価値

「同じことを別の技術で実装する」だけでは通常WPの上位互換にはならない。むしろ通常WPの方が早く・安く・確実に作れる。

Headlessの本当の価値は、**通常WPでは原理的に難しい、または不可能なこと**ができる点:

1. **マルチプラットフォーム配信**: 同じデータをWeb + モバイルアプリ + LINE Mini App + サイネージに配信
2. **モダンなUI/UX**: SPA体験、リアルタイム更新、複雑なインタラクション
3. **エッジ配信による圧倒的速度**: Vercel Edge等で世界中に静的配信
4. **複数データソースの統合**: WP + Stripe + Supabase + Algolia等を1つのフロントに統合
5. **フロントとバックの独立デプロイ**: チーム分業しやすい
6. **モダン開発体験**: TypeScript、自動テスト、CI/CD、Git運用が前提

## メリット・デメリット

### メリット

1. 表示速度が圧倒的に速い（SSG・ISRで高速化可能）
2. セキュリティが強化される（管理画面と公開サイトを分離）
3. デザイン自由度が無制限
4. マルチプラットフォーム対応（同じデータをWeb・アプリ・LINEで使える）
5. SEOに強い（適切に実装すれば高速＋構造化データで有利）
6. ポートフォリオ映えする（モダン技術スタックで採用評価が高い）

### デメリット

1. 初期構築コストが高い（通常WPの2〜3倍の工数）
2. 構成が複雑（WP・API・フロントエンドの3層構造）
3. プレビュー機能の実装が面倒（編集者体験の作り込みが必要）
4. 編集者の戸惑い（従来のWPとは操作感が違う）
5. 学習コストが高い（複数の技術を組み合わせる）
6. 画像最適化等で詰まりがち

## 必要なプログラミング言語

### 🔴 必須（フロントエンド側）

| 技術 | 役割 | 学習目安 |
|------|------|---------|
| HTML | ページ構造の記述 | 基礎は前提 |
| CSS | スタイリング | 基礎 + Tailwind CSS |
| JavaScript（ES6+） | 全動作の中核言語 | 中級レベルまで必須 |
| React | UIコンポーネントライブラリ | フック理解必須 |
| Next.js | フロントエンドフレームワーク | App Router |

**JavaScriptが最も重要**。Headless WordPressの本体は「JavaScriptでWordPressのAPIを叩いて表示するアプリ」なので、ES6以降の構文（アロー関数、分割代入、async/await、配列メソッド等）は必須。

### 🟡 強く推奨

| 技術 | 役割 | 理由 |
|------|------|------|
| TypeScript | JavaScriptに型安全性を追加 | 案件の9割で必須 |
| Tailwind CSS | ユーティリティファーストCSS | Next.js案件のデファクト |

### 🟢 必要に応じて学ぶ（バックエンド側）

| 技術 | 役割 | いつ必要か |
|------|------|----------|
| PHP | WordPress本体・プラグインカスタマイズ | カスタムフィールド追加、認証実装時 |
| SQL | データベース直接操作 | 大規模サイトの最適化、データ移行時 |

PHPは「読めれば良い」レベルでスタートして十分。

### 🛠 周辺ツール

- Git / GitHub（バージョン管理）
- Node.js v22 LTS（最低v20以上）
- npm / pnpm（パッケージ管理）
- VSCode / Cursor（エディタ）
- Vercel（デプロイ先）

## 実際の言語比率

Headless WordPress案件で書くコードの内訳:

```
TypeScript / JavaScript : 70%
CSS（Tailwind含む）      : 15%
HTML（JSX内）            : 10%
PHP（WP側カスタマイズ）  :  5%
```

メインで学ぶべきは **TypeScript + React + Next.js**。WordPressは「裏方の道具」として使うイメージ。
