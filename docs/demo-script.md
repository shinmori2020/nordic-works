# デモ動画 録画台本

公開URL（Vercel 静的サイト）では動作しない動的機能を、ローカル環境で証明するための録画用台本。

**前提環境**:
- Local の `nordic-works` サイトが起動中
- `web/.env.local` に Application Password と secret 設定済み
- `pnpm dev` で `http://localhost:3000` が起動中
- ブラウザは Chrome（拡張機能アイコンを最小限に）

**録画ツール推奨**:
- 簡単に撮るなら **Windows Game Bar**（`Win + G`）
- 編集や注釈を入れるなら **OBS Studio**（無料）
- ナレーション込みなら **Loom**（ブラウザ完結）

**録画前のチェックリスト**:
- [ ] ブラウザのブックマーク・履歴サジェスト・拡張機能のアイコンを最小化
- [ ] 通知（Slack/メール）を OFF
- [ ] 1080p 以上で録画（YouTube 推奨）
- [ ] フォントサイズを 1段大きく（視認性）
- [ ] WP 管理画面に**ログイン済み**の状態で開始
- [ ] 不要なタブを閉じる

## 🎬 動画 1: 下書きプレビュー機能（目標 2〜3分）

### タイトル候補
- 「Headless WordPress で下書きプレビュー — Next.js 16 + draftMode」
- 「ブロックエディタからの下書きプレビュー（Headless WP デモ）」

### シーン構成

#### シーン 1 — オープニング（10秒）
**画面**: `https://nordic-works.vercel.app/articles`
**ナレーション/字幕**:
> Nordic Works — Headless WordPress で作ったポートフォリオです。今回は **下書きプレビュー機能**を実演します。

#### シーン 2 — 問題提起（15秒）
**画面**: 公開URLのまま、適当な記事スラッグを編集して `/articles/<存在しない>` にアクセス → 404 を見せる
**ナレーション/字幕**:
> 下書きは公開URLからは見えません（404）。当然、検索エンジンにも公開してはいけません。
> 編集者は**公開前に Next.js 上で見た目を確認**したい。これを実装します。

#### シーン 3 — WP 管理画面で下書き作成（30秒）
**画面**: `http://nordic-works.local/wp-admin` → 投稿 → 新規追加
**操作**:
1. タイトルを入力（例: `デモ用の下書き記事`）
2. 本文を 1〜2 段落書く（コピペで OK）
3. **「下書き保存」**をクリック

**ナレーション/字幕**:
> WP 管理画面で新規記事を作成し、**下書き保存**します。

#### シーン 4 — プレビューボタン押下 → Next.js へ遷移（20秒）
**画面**: WP 編集画面 → 右上の **「プレビュー」→「新しいタブでプレビュー」** ボタン
**操作**:
1. プレビューボタンをクリック
2. 新しいタブが開き、`localhost:3000/articles/<slug>` に遷移
3. **画面上部に黄色いバナー**「プレビュー表示中（下書きを含みます）」を指差し

**ナレーション/字幕**:
> プレビューボタンをクリックすると、**Next.js 側で下書きが表示**されます。
> ヘッダー上の黄色いバナーが「プレビューモード中」を示します。
> 下書きの本文・タイトルが見えています。

#### シーン 5 — 設計のポイント解説（30秒）
**画面**: VS Code でファイルを2つだけ見せる
1. `web/src/app/api/preview/route.ts`（特に `draftMode().enable()` と secret 検証部分）
2. `app/public/wp-content/plugins/nordic-works-core/nordic-works-core.php` の `nordic_frontend_link` 関数

**ナレーション/字幕**:
> 仕組みは3層:
> ① WP プラグインの `post_link` フィルターが REST 上のリンクを `/api/preview` に書き換え
> ② Next.js の `/api/preview` が secret 検証して `draftMode` を有効化
> ③ 各ページで `fetchBySlug` が draft モードを検出し、認証付きで下書きを取得
>
> ブロックエディタは PHP の `preview_post_link` フィルターを通らないため、`post_link` 側で REST の `link` 自体を書き換えるのがコツです。

#### シーン 6 — プレビュー終了（15秒）
**画面**: バナーの **「プレビューを終了」** リンクをクリック
**操作**:
1. リンククリック → トップへリダイレクト
2. 再度同じ URL を直打ち → **404** に戻る

**ナレーション/字幕**:
> 「プレビューを終了」で `draftMode` を無効化。Cookie が削除されて、通常モードに戻ります。
> 直接アクセスは再び 404 — **公開前の記事が漏洩しない設計**です。

#### シーン 7 — クロージング（10秒）
**画面**: README または GitHub の該当箇所
**ナレーション/字幕**:
> 実装の詳細は GitHub で公開しています。ご覧いただきありがとうございました。

## 🎬 動画 2: On-demand Revalidation（目標 2〜3分）

### タイトル候補
- 「WordPress 更新 → Next.js 即時反映 — On-demand Revalidation」
- 「ISR を超えるリアルタイム反映（Headless WP デモ）」

### シーン構成

#### シーン 1 — オープニング & 問題提起（20秒）
**画面**: `http://localhost:3000/articles` で既存記事カードを映す
**ナレーション/字幕**:
> 通常の ISR は「24時間ごとに再生成」のような時間ベース。
> 編集者からは「**今すぐ反映したい**」というニーズが必ず出ます。
> WP の更新と同時にフロントへ自動反映する仕組みを実装しました。

#### シーン 2 — 反映前の状態を見せる（20秒）
**画面**:
1. `localhost:3000/articles` を開いて、テスト対象記事のタイトルを画面に映す
2. 画面分割で `nordic-works.local/wp-admin` の同じ記事の編集画面も開く

**ナレーション/字幕**:
> 左が Next.js、右が WP 管理画面です。同じ記事を映しています。

#### シーン 3 — タイトル変更 & 公開（30秒）
**画面**: WP 編集画面
**操作**:
1. タイトル末尾に `（更新済み）` などを追記
2. **「更新」** ボタンをクリック
3. 直後（数秒以内）に Next.js タブをリロード
4. **タイトルが新しい値**に変わっていることを指差し

**ナレーション/字幕**:
> タイトルを書き換えて「更新」。
> Next.js 側をリロードすると…**即座に反映**されています。
> 24時間待つ必要がありません。

#### シーン 4 — 開発者ツールで webhook を確認（30秒）
**画面**:
1. WP 編集画面でもう一度タイトルを少し変えて「更新」
2. **Next.js dev サーバーのターミナル**を画面に表示
3. ターミナルログに `POST /api/revalidate 200` が出るのを見せる

**ナレーション/字幕**:
> 仕組みは、WordPress の `save_post` フックから **webhook を投げる**だけ。
> ターミナルに `POST /api/revalidate 200` が出ているのが見えます。
> このリクエストが Next.js のキャッシュをクリアしています。

#### シーン 5 — コード解説（40秒）
**画面**: VS Code で3ファイルを順に映す
1. `app/public/wp-content/plugins/nordic-works-core/nordic-works-core.php` の `nordic_trigger_revalidate` 関数
2. `web/src/app/api/revalidate/route.ts`（`revalidatePath` + `revalidateTag('tag', 'max')`）
3. `web/src/lib/wordpress.ts` の `tags: ['posts', \`post-${slug}\`]` 部分

**ナレーション/字幕**:
> ① WP プラグインが `wp_remote_post` で `blocking=false`（非同期）で webhook 送信
> ② Next.js 側は secret token 検証後、投稿タイプに応じた `revalidatePath` と `revalidateTag('tag', 'max')` を実行
> ③ 取得関数側で各 fetch に `tags` を付けているので、タグ単位で正確にクリアできる
>
> Next.js 16 で `revalidateTag` のシグネチャが変わり第2引数必須化されたので、`'max'` を渡しています。

#### シーン 6 — 公開URLでは動作しない理由（20秒）
**画面**: Vercel ダッシュボードのデプロイ画面
**ナレーション/字幕**:
> 本構成は **Local WordPress + Vercel 静的サイト**のため、本番では webhook が到達しません。
> 本番 WordPress（Xserver 等）に移行すれば、コード変更ほぼなしで本番反映可能です。

#### シーン 7 — クロージング（10秒）
**ナレーション/字幕**:
> 詳細は GitHub をご覧ください。ありがとうございました。

## 録画後のチェックリスト

- [ ] 動画の冒頭/末尾の不要な部分をトリミング
- [ ] 個人情報（メールアドレス・トークン等）が映り込んでいないか確認
  - `WP_APPLICATION_PASSWORD` / `RESEND_API_KEY` / `ALGOLIA_ADMIN_KEY` が画面に出ていたら必ずモザイク or 再録画
- [ ] タイトル・サムネ・概要欄を YouTube にアップ
  - 概要欄テンプレートは [`youtube-descriptions.md`](./youtube-descriptions.md) を使用
- [ ] **公開設定は「限定公開（unlisted）」**にする（URL を知る人のみ視聴可）
- [ ] 取得した動画 ID（`https://youtu.be/<ここ>`）を README の `YOUR_VIDEO_ID_1` / `YOUR_VIDEO_ID_2` に差し替え

## 録画の心理的ハードルを下げるコツ

- 1テイクで完璧を目指さず、**シーンごとに分けて撮って結合**でもOK
- ナレーションは録画後に**音声を別収録して上乗せ**してもOK（録画中は無言で操作だけ）
- 「ます/です」が言いにくければ**字幕オンリー**でも全く問題なし
- 1本目を撮ってみて学んでから 2本目、の順番が楽
